# AGENTS.md

## 1. 项目定位

本项目是一个软件开发项目。Codex 在执行任务时，需要先理解项目结构、运行方式、代码规范和验证方式，再进行修改。

在开始任何编码前，请先快速阅读以下内容：

* README.md
* package.json / pyproject.toml / requirements.txt / pom.xml / go.mod 等依赖配置文件
* src/ 或 app/ 目录下的核心代码
* tests/ 目录下的测试用例
* docs/ 目录下的说明文档，如果存在

如果项目结构不清晰，请先总结你看到的目录结构，再开始修改。

---

## 2. 工作原则

Codex 执行任务时请遵守以下原则：

1. 先理解，再修改，不要直接大面积重写。
2. 优先做最小可行修改，避免无关改动。
3. 不要随意删除已有功能、注释、配置文件或测试代码。
4. 不要擅自更换技术栈、框架、目录结构。
5. 不要引入新的生产依赖，除非任务明确要求；如果必须引入，请说明原因。
6. 修改代码后，需要说明改了哪些文件、为什么这么改、如何验证。
7. 如果需求有歧义，先列出你的理解和假设，再继续。
8. 对于复杂任务，先给出计划，再分步骤执行。

---

## 3. 代码风格

请尽量保持与项目现有代码风格一致。

通用要求：

* 变量命名清晰，不要使用无意义命名。
* 函数只做一件相对明确的事情。
* 避免重复代码，必要时抽取公共函数。
* 保持代码可读性，不要为了炫技写复杂实现。
* 重要逻辑需要添加简洁注释。
* 不要添加大量无意义注释。
* 不要留下调试代码，例如 print、console.log、debugger，除非项目本身需要。

---

## 4. Git 与文件修改规则

执行任务时请注意：

* 不要修改与当前任务无关的文件。
* 不要修改 .env、密钥、token、证书等敏感文件。
* 不要把真实 API Key、密码、Cookie 写入代码或文档。
* 不要自动提交 git commit，除非我明确要求。
* 不要自动 push 到远程仓库，除非我明确要求。
* 如果发现已有未提交改动，不要覆盖，先说明情况。
* 修改前后尽量保持项目可以正常运行。

---

## 5. 运行与验证

完成修改后，请根据项目类型选择合适的验证方式。

如果是 Python 项目，优先尝试：

```bash
python -m pytest
python app.py
python main.py
```

如果是 Node.js / 前端项目，优先尝试：

```bash
npm install
npm run dev
npm run build
npm run lint
npm test
```

如果是后端项目，优先查看 README.md 或配置文件中的启动方式。

验证时请遵守：

* 不确定命令时，先查看 README.md、package.json、pyproject.toml 等文件。
* 不要盲目运行危险命令。
* 如果测试失败，请区分是你修改导致的失败，还是项目原本就存在的问题。
* 最终回复中需要说明是否运行了测试，以及测试结果。

## 6. 任务执行流程

对于普通任务，请按以下流程执行：

1. 理解需求。
2. 查看相关文件。
3. 给出简短修改计划。
4. 修改代码。
5. 运行必要检查。
6. 总结改动和验证结果。

对于复杂任务，请先创建清晰计划：

* 目标是什么
* 涉及哪些模块
* 需要改哪些文件
* 有哪些风险
* 如何验证完成

未经确认，不要一次性重构大量代码。

---

## 7. 前端项目规则

如果本项目是前端项目，请遵守：

* 保持组件职责清晰。
* 不要把复杂业务逻辑全部写在页面组件里。
* 优先复用已有组件。
* 样式风格要和现有页面保持一致。
* 不要随意更改全局样式。
* 表单、按钮、弹窗、错误提示要考虑基本用户体验。
* 涉及接口调用时，需要考虑 loading、error、empty 状态。

---

## 8. 后端项目规则

如果本项目是后端项目，请遵守：

* 接口逻辑要清晰，避免把所有逻辑写在路由层。
* 输入参数需要做基本校验。
* 错误处理要清楚，不要吞掉异常。
* 不要在日志中输出密码、token、隐私数据。
* 数据库操作要注意事务、重复提交和异常回滚。
* 新增接口时，尽量补充接口说明或测试用例。

---

## 9. AI / Agent 项目规则

如果本项目涉及 AI、LLM、Agent、RAG 或 LangChain，请遵守：

* Prompt 要单独管理，避免散落在业务代码中。
* Prompt 修改后要说明修改目的。
* 不要把 API Key 写死在代码里。
* 模型调用需要考虑异常、超时、重试和空结果。
* RAG 项目需要区分：数据加载、切分、向量化、检索、生成、评估。
* Agent 项目需要明确：工具列表、工具输入输出、调用边界、失败处理。
* 不要让 Agent 执行危险命令或访问无关文件。
* 涉及评测时，需要尽量保留测试样例和评测指标。

## 11. Codex 工作规则


本项目禁止一次性黑盒式完成大任务。

每次 Codex 执行任务必须遵守：

1. 任何“完成”都必须以可运行、可验证为准，不允许只创建空文件后声明完成。
2. 每次开始前先读取 docs/CODEX_STATUS.md。
3. 每次结束前必须更新 docs/CODEX_STATUS.md。
4. 每阶段最多修改 3-5 个文件。
5. 每阶段必须说明：

   * 本阶段完成了什么；
   * 修改了哪些文件；
   * 如何启动或验证；
   * 浏览器中应该看到什么效果；
   * 剩余未完成事项。
6. 如果 token、额度、上下文或工具中断，必须把当前状态标记为 partial，不得标记为 complete。
7. 不要自动 commit，不要自动 push，除非用户明确要求。
8. 多 Agent 只在用户明确要求时启用。
9. 默认使用低 token 模式：少解释、不粘贴大段代码、优先更新状态文件。
